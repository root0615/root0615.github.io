## ✒️ Dockerfile에 대해
Dockerfile은 베이스가 되는 OS 그리고 어떤 환경을 사용할지 정하고 실행하기 위해 여러가지 작업 명령어를 써놓은 파일입니다.<br>
그럼 Dockerfile에서 사용하는 명령어를 알아보겠습니다.<br>

### 1. FROM -- 베이스 이미지 지정
```
FROM python:3.11-slim
```
Dockerfile에서 반드시 첫 번째 줄.<br>
컨테이너의 출발점이 되는 이미지를 지정합니다. 그러니까 `어떤 OS와 기본환경 위에 내 앱을 올릴지 결정`합니다.

📌 다른 예시<br>
```
FROM python:3.12-slim
FROM ubuntu:22.04       # 우분투 베이스(파이썬 직접 설치 필요)
```

❓Slim 과 기본의 차이?<br>
python:3.11은 약 900MB, python:3.11-slim은 약 130MB입니다.<br>
slim은 불필요한 도구들이 제거된 버전으로, 특별한 이유가 없으면 slim으로 쓰는게 좋습니다.<br>

### 2. WORKDIR -- 작업 폴더 설정
```
WORKDIR /app
```
컨테이너 안에서 작업할 기본 폴더를 지정합니다.<br>
폴더가 없으면 자동으로 만들어줍니다.<br>
이후 COPY, RUN, CMD 등 모든 명령어가 이 경로를 기준으로 실행됩니다.<br>

📌 WORKDIR 없이 쓰면<br>
```
COPY main.py main.py    # /main.py에 저장됨 (루트에 쌓임)
RUN pip install...      # 위치 불분명
```

📌 WORKDIR 쓰면<br>
```
WORKDIR /app
COPY main.py .      # /app/main.py에 저장됨 (명확)
```

📌 왜 /app 인가?<br>
/app은 관례적으로 많이 쓰는 경로이며, /code, /srv 등 아무 이름이나 써도 괜찮습니다.<br>
루트(/)에 파일을 바로 쌓으면 시스템 파일과 섞여서 위험하기에 분리해주는 역할을 합니다.

### 3. COPY -- 파일을 컨테이너로 복사
```
# 문법: COPY [내 컴퓨터 경로] [컨테이너 경로]
COPY requirements.txt .   # 파일 하나
COPY . .                  # 현재 폴더 전체
```
내 컴퓨터의 파일을 컨테이너 안으로 복사합니다.<br>
점(.)은 '현재 위치'들 뜻하며, 첫번째(.)는 내 컴퓨터의 현재 폴더, 두번째(.)는 컨테이너의 WORKDIR(/app)입니다.


📌 순서가 중요<br>
```
COPY requirements.txt .     # 먼저 설치할 패키지 목록 가져오기
RUN pip install ...         # 패키지 설치
COPY . .                    # 나머지 코드 (자주 바뀜)
```

📌 requirements.txt를 먼저 복사하는 이유<br>
Docker는 레이어 캐시를 사용합니다.<br>
requirements.txt가 바뀌지 않으면 pip install 레이어를 재사용해서 빌드 시간이 크게 줄어듭니다.<br>
코드만 바꿨을 때 패키지를 다시 설치 하지 않아도 됩니다.<br>
위의 설명만으로는 초보 입장에선 왜 그렇게 되는지 모릅니다.(제가 모릅니다.)<br>
그래서 설명하자면, Docker의 레이어 캐시 개념을 이해하면 이해할 수 있습니다.<br>
![도커의 레이어 캐시](/assets/img/docker/Docker_layer.png)<br>
[출처] claude가 만들어줌<br>

> 핵심 규칙은 Docker는 어떤 레이어가 바뀌면 그 아래 레이어는 전부 캐시를 버리고 다시 실행합니다.<br>
레이어는 명령어 한 줄 단위로 움직입니다.<br>
그래서 자주 바뀌는 것(코드)은 아래에, 잘 안 바뀌는 것(패키지 목록)은 위에 두는 겁니다.<br>
위 이미지 예시와 같이 main.py만 바뀌었을 때 설치할 패키지 목록을 먼저 가져와서 인스톨 명령을 진행하고, 전부 복사하면, 가져오고 설치하는 과정을 건너뛸 수 있다는 얘기입니다.<br>
물론 처음에는 설치를 다하겠죠? 이건 그러니까 반복적으로 빌드할 때 이야기입니다.<br>
개발할 때 고치고 빌드, 테스트. 고치고 빌드, 테스트할 때 과정을 반복하니 매번 pip install이 새로 돌면 너무 느리니까 순서를 지키는 겁니다.<br>
제가 이해하기 쉽게 설명했는지 잘모르겠네요...<br>

### 4. RUN -- 빌드 중 명령어 실행
이미지를 만들 때 딱 한번 실행됨<br>
```
RUN pip install --no-cache-dir -r requirements.txt

# 여러 명령어는 &&로 합치기 (레이어 절약)
RUN apt-get update && apt-get install -y curl
```
docker build 중에 실행되는 명령어입니다.<br>
패키지 설치, 파일 생성, 설정 등을 여기서 해요.<br>
RUN은 이미지 레이어를 하나 만들기 때문에 명령어가 많으면 &&로 묶는 게 좋습니다.<br>

📌 RUN을 여러번 쓰면 (비추)<br>
```
RUN apt-get update       # 레이어 1
RUN apt-get install curl # 레이어 2 (불필요하게 늘어남)
```

📌 &&로 합치면 (추천)<br>
```
RUN apt-get update && apt-get install -y curl  # 레이어 1개
```

📌 --no-cache-dir의 의미<br>
pip가 다운로드한 파일을 캐시로 저장하지 않도록 합니다.<br>
컨테이너 안에서는 캐시가 필요없기 때문에 이미지 용량을 줄이는 효과가 있습니다.<br>

### 5. EXPOSE -- 포트 번호 선언
실제 연결은 docker run -p에서
```
# EXPOSE [포트번호]
EXPOSE 8000
```
이 컨테이너가 어떤 포트를 사용하는지 '문서로 기록'하는 역할을 합니다.<br>
실제로 포트를 열어주는 건 아닙니다.<br>
실행 할 때 docker run -p 8000:8000 을 해야 외부에서 접근이 가능합니다.<br>

📌 EXPOSE 없어도 동작은 합니다.<br>
EXPOSE는 필수가 아닙니다만, Dockerfile을 읽는 사람에게 '이 앱은 8000번 포트를 씁니다.'라고 알려주는 문서 역할을 하기에 쓰는게 관례라고 합니다.<br>

### 6. CMD -- 컨테이너 시작 시 실행
docker run 할 때마다 실행되는 명령어
```
# 올바른 형식 (exec form — 항상 이 형식 사용)
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

# 잘못된 형식 (shell form — 쓰지 마세요)
CMD uvicorn main:app --host 0.0.0.0 --port 8000
```
컨테이너가 시작될 때 실행할 명령어입니다.<br>
RUN은 빌드 중에 실행되고, CMD는 실행(run)할 때마다 실행됩니다.<br>
배열 형식(exec form)으로 쓰는 것이 공식 권장사항입니다.<br>

📌 CMD vs RUN<br>
RUN  → 이미지 빌드 중 실행 (pip install 등 준비 작업)<br>
CMD  → 컨테이너 시작할 때마다 실행 (앱 구동)<br>

📌 --host 0.0.0.0을 하는 이유<br>
기본값(127.0.0.1)은 컨테이너 내부에서만 접근이 가능합니다.<br>
0.0.0.0으로 해야 외부(내 브라우저)에서 접근할 수 있습니다.<br>
이걸 빠뜨리면 포트를 열어도 연결이 안됩니다.<br>

### 7. ENV -- 환경변수 설정
설정값 주입
```
ENV APP_ENV=production
ENV PORT=8000

# 실행 시 덮어쓰기 가능
# docker run -e APP_ENV=development ...
```
컨테이너 안에서 사용할 환경변수를 설정합니다.<br>
Python 코드에서는 os.getenv('APP_ENV')로 읽을 수 있습니다.<br>
API 키 같은 민감한 정보는 Dockerfile에 직접 쓰지 말고 docker run -e 로 주입합니다.<br>

⚠️ 민감한 정보는 절대 Dockerfile에 쓰지 않습니다.<br>
Dockerfile을 공유할게 아니라면 상관없지만, 보통 공유할 경우 git에 올리는데 잘못 설정하여 Dockerfile이 올라가면 대참사가 나게 되겠지요...<br>

### 📝 전체 Dockerfile 흐름 정리
```
# 예시 
FROM python:3.11-slim        # ① 베이스 환경 준비
WORKDIR /app                 # ② 작업 폴더 지정
COPY requirements.txt .      # ③ 패키지 목록 복사
RUN pip install ...          # ④ 패키지 설치 (캐시됨)
COPY . .                     # ⑤ 전체 코드 복사
EXPOSE 8000                  # ⑥ 포트 문서화
CMD ["uvicorn", "main:app",  # ⑦ 실행 명령어 등록
     "--host", "0.0.0.0",
     "--port", "8000"]
```
FROM 명령어 ~ EXPOSE 명령어까지는 docker build할 때 실행.<br>
CMD 명령어는 docker run 할 때마다 실행되는 점이 핵심입니다.<br>

## ✒️ Docker 이미지에 대해
이미지는 한마디로 "컨테이너를 찍어내는 틀"입니다.<br>
붕어빵에 비유하자면, Dockerfile은 붕어빵 레시피이고, 이미지는 붕어빵 틀, 컨테이너는 거기서 찍어낸 붕어빵입니다.<br>
틀 하나로 붕어빵을 여러개 찍을 수 있듯이, 이미지 하나로 컨테이너를 여러개 만들 수 있습니다.<br>

### 📝 이미지의 중요한 특징 2가지<br>
1. 읽기 전용<br>
이미지 자체는 절대 바뀌지 않습니다.<br>
컨테이너를 실행하면 이미지 위에 얇은 읽기/쓰기 레이어가 하나 추가되고, 컨테이너 안에서 파일을 만들거나 바꾸는건 그 레이어에만 기록됩니다.<br>
이미지 원본은 그대로예요.<br>

2. 이미지 하나로 여러 컨테이너 생성<br>
같은 이미지에서 컨테이너를 3개 띄우면 트래픽을 나눠서 받을 수 있습니다.<br>
이게 스케일 아웃의 기초가 된다고 합니다.<br>


## ✒️ Docker 컨테이너에 대해
한마디로 "내 앱이 혼자 돌아가는 격리된 미니 컴퓨터"입니다.

### 📝 핵심 특징 3가지
1. 격리<br>
컨테이너A에서 뭔가 설치하거나 망가뜨려도 컨테이너B와 내 컴퓨터에는 전혀 영향이 없습니다.
2. 가벼움<br>
VM(가상머신)은 Windows나 Linux OS 전체를 통째로 흉내 내서 많은 용량에 부팅도 오래걸리지만, 컨테이너는 Host OS의 커널을 공유하기 때문에 수백MB 수준이고 몇초만에 실행됩니다.
3. 이식성<br>
컨테이너는 이미지에서 만들어지기 때문에 내 컴퓨터든, 회사 서버든, AWS든 어디서나 똑같이 실행됩니다.
